# ADR: Recipe Schema

## Status

Accepted.

## Context

The recipe feature is currently mock-backed. `src/pages/recipes/recipes.data.ts` defines the list/detail shape, filtering, and last-viewed sort behavior. `src/pages/recipes/recipes.api.ts` exposes server functions that return mock data. `src/routes/api/recipes/upload.ts` validates the add form and optimizes images, but it only writes temporary files and does not persist recipes or upload metadata yet.

The existing database schema contains Better Auth tables and `upload_object`. User IDs and upload object IDs are text. Recipe tables will be the first app-domain tables. The app is a solo hobby app, so the initial schema should prefer low-friction persistence over premature normalization.

The current UI implies these concepts:

- Recipes are managed by an authenticated owner.
- The recipe list searches name, tags, and description.
- The recipe list filters by kcal and rating.
- Ingredients are displayed as an ordered list of strings.
- Tags are displayed as strings.
- Recipes can have up to eight ordered photos.
- Nutrition values are optional.
- Detail pages have recipe management actions and a rating affordance.
- Signed-in recipe ordering should be able to prioritize recently viewed recipes.

## Decision

Create recipe-domain tables in `src/db/schema/recipes.schema.ts`:

- `recipe`
- `recipe_image`
- `recipe_last_view`

Recipes are private per user. Each recipe has a required `user_id` foreign key to `user.id`, with `ON DELETE CASCADE` so deleting a user removes their recipes.

Recipe IDs and recipe image IDs use Postgres UUIDs generated by the database with PostgreSQL 18's built-in `uuidv7()` function. UUIDv7 keeps IDs time-sortable while preserving UUID semantics. Foreign keys to existing `user` and `upload_object` tables keep their existing text types.

Store ingredients and tags as Postgres `text[]` columns on `recipe`. This keeps the first implementation simple while preserving ingredient order and matching the current UI. Do not add normalized ingredient or tag tables until the app needs reusable ingredients, tag pages, tag-level analytics, or stronger tag querying.

Store one owner rating on `recipe.rating`. This represents the creator's 1-5 rating, not an aggregate and not per-user ratings. The detail page's current "Your rating" mock should be reconciled during implementation so it edits or displays this owner rating instead of implying separate user ratings.

Store nutrition on `recipe` as optional per-serving integer fields:

- `kcal`
- `protein`
- `carbs`
- `fats`

Also store `portions` as a required positive integer. `portions` is how many servings the listed ingredients produce. Macro fields are per portion/serving, not for the whole recipe.

Store photos with `recipe_image` rows. Each row points to one `upload_object.id` and records `position` for display order. `upload_object` remains the source of storage bucket/key/mime metadata.

Add `recipe_last_view` keyed by `(user_id, recipe_id)` with `viewed_at`. This supports the existing mock behavior where a signed-in user's recently viewed recipes sort ahead of the default updated-at ordering.

Use minimal indexes for the initial implementation:

- recipe owner lookup
- recipe image lookup by recipe

When entering the recipe page, the app will fetch all recipes for the current user. The expected recipe count per user is small, so sorting and filtering by updated date, kcal, rating, name, tags, and description should happen client-side for now. Do not add full-text search, trigram indexes, filter indexes, sort indexes, or Postgres extensions until the search experience proves it needs them.

## Delete Behavior

Deleting a recipe should delete its `recipe_image` rows by database cascade.

Deleting the related `upload_object` metadata and physical R2 objects cannot be fully modeled by database cascades, because R2 is outside the database and because the relationship points from `recipe_image` to `upload_object`. The recipe delete application flow should collect the related upload objects, delete the R2 objects, delete or mark the upload metadata as removed, then delete the recipe in a transaction where possible.

`recipe_image.upload_object_id` is unique so one upload object belongs to at most one recipe image. That keeps orphan cleanup understandable.

## Consequences

This schema keeps the first persisted recipe implementation small and close to the current UI. It avoids creating normalized tables for ingredients, tags, or per-user ratings before there are product requirements for them.

The tradeoff is that tag search and ingredient search are not strongly relational. If recipes grow into a larger catalog, tags and ingredients may need to be extracted into child tables later. A future family-sharing feature may also require ownership, sharing, and indexing changes, but it is intentionally out of scope for this schema.

UUID recipe IDs differ from the existing text primary keys used by auth and upload tables. This is acceptable for new domain tables, but route params and server-function validators should validate recipe IDs as UUID strings before querying.

R2 cleanup remains an application concern. Tests for recipe deletion should cover both database rows and storage cleanup once persistence is implemented.
