Harden before public exposure

Strong data boundaries. Two abuse paths need closing.

The app correctly isolates each user’s diary and recipes, and no direct account takeover, SQL injection, XSS sink, or cross-user data access was found. The main internet risk is that anyone can register and then exercise an expensive upload path without a quota.

02 / Go-live plan

Do these three things first

These changes provide most of the security value without introducing heavyweight infrastructure. They fit a single-owner hobby deployment.

  1. Close account creation after bootstrapping the owner

    Disable email/password and Google signup, or enforce a short email allowlist. If public signup is intentional, require verification and add per-user quotas.

    Small change
  2. Bound uploads before Node parses them

    Add a proxy body cap, aggregate byte and field limits, a realistic pixel ceiling, plus simple per-user/IP throttling and storage quotas.

    Small–medium
  3. Make production security explicit

    Require HTTPS app origin and a non-placeholder secret, set production mode, enable auth rate limiting and secure cookies, then add frame/cache/security headers at the app or edge.

    Small change

13 findings shown

H-01 · Access policy

Unrestricted signup turns a personal app into a public service

Email/password accounts auto-sign in without verification, and Google can also create new users. That is a policy mismatch for a personal deployment.

High

Why it matters

Any visitor can obtain an authenticated session, create unlimited rows, and use the image-processing/R2 path. User isolation still holds, but compute, storage, database growth and email namespace are available to strangers.

Evidence

  • src/lib/auth/auth.ts:23–34
  • src/pages/signup/SignupPage.tsx:104–110
  • Better Auth defaults disableSignUp to false.

Low-friction fix

Create the owner account, then set disableSignUp: true for credentials and Google and remove/guard /signup. For a few trusted users, an email allowlist is simpler than building invitations.

Acceptance test

A fresh email and a new Google identity both fail to create an account, while the existing owner can still sign in. If signup remains public, require verified email and verify per-user row/storage quotas instead.

H-02 · Availability and cost

The upload route can be used as an 80 MiB CPU and storage amplifier

The entire multipart request is parsed before size checks, then up to eight 10 MiB files are decoded and re-encoded. No route throttle, aggregate quota, or tight image complexity limit exists.

High

Why it matters

An account can repeatedly occupy Node memory and Sharp CPU, generate R2 traffic, and grow storage/database state. Open signup makes obtaining that account trivial. Highly compressed, high-pixel inputs increase decode cost.

Evidence

  • src/routes/api/recipes/upload.ts:13–14, 50–69
  • src/routes/api/recipes/upload.ts:80–95, 198–215
  • Text/array bounds are incomplete at lines 28–40.

Low-friction fix

Enforce a body cap at the edge before parsing; add total photo bytes, field/item lengths and counts in ArkType; constrain nonnegative integers; set a practical limitInputPixels and stricter failOn; add per-user/IP write limits and a storage quota.

Acceptance test

Oversized and chunked bodies are rejected by the proxy before Node allocates the multipart payload; high-pixel/corrupt images fail with 400; repeated requests reach 429; and a user cannot exceed the chosen storage/row allowance.

M-01 · Production configuration

Auth safety depends on deployment values the image does not enforce

The runner does not set production mode, while the app origin falls back to http://localhost:3000. Better Auth’s limiter and cookie behavior are environment-sensitive.

Medium

Why it matters

If Dokploy does not inject the expected values, credential throttling is off, the trusted auth origin is wrong, browser calls may target localhost, and auth cookies may lack Secure. This is a deployment-dependent failure mode.

Evidence

  • Dockerfile:3–9, 22–30
  • src/lib/env/server.ts:1–15
  • src/lib/env/client.ts:1–5
  • src/lib/auth/auth.ts:10–12

Low-friction fix

Set NODE_ENV=production in the runner, explicitly enable auth rate limiting, use same-origin browser auth, and require a separate valid HTTPS APP_URL in production. Force secure cookies and configure trusted proxy IP headers deliberately.

Acceptance test

The deployed bundle contains no localhost origin; session cookies include Secure; HttpOnly; SameSite=Lax; foreign origins are rejected; and rapid login attempts return 429 using the real client IP.

M-02 · Secret validation · Conditional

The public example auth secret is accepted as production-valid

The setup guide copies an example containing a known 41-character placeholder, but startup only checks that the variable exists. The current local value differs; the live Dokploy value was not available.

Medium

Why it matters

A forgotten placeholder is public knowledge and undermines signed auth cookies, OAuth state and verification tokens. Its length means library strength checks do not recognize it as a placeholder.

Evidence

  • .env.example:8
  • README.md:9–10
  • src/lib/env/server.ts:1–12

Low-friction fix

Leave the example value empty, document a cryptographic generation command, and validate the environment with ArkType at startup: minimum strength plus explicit rejection of known placeholders. Rotate production if it ever used the example.

Acceptance test

Production startup fails on empty, short, or known-placeholder secrets and succeeds with a newly generated value. Secret values never appear in logs, build output, or Docker build arguments.

M-03 · Media privacy · Conditional

Recipe authorization does not protect the direct photo URL

Authenticated recipe reads return a public CDN/R2 URL. Random UUIDs prevent guessing, but anyone who receives a URL can fetch the photo without a session.

Medium

Why it matters

URLs can escape through sharing, history, logs, referrers or CDN access records. The key also contains a stable user ID. This is fine only if “public by unguessable link” is the intended privacy model.

Evidence

  • src/routes/api/recipes/upload.ts:82–92
  • src/lib/storage/config.ts:15–22
  • src/pages/recipes/recipes.api.ts:145–165

Low-friction fix

If photos are private, keep R2 private and return short-lived signed URLs, or use an authenticated media endpoint that checks recipe ownership. Remove user IDs from object keys. Otherwise document and accept the capability-URL model.

Acceptance test

A copied image URL fails without authorization or expires quickly; bucket listing is disabled; deleted content is no longer retrievable; and CDN caching matches the chosen privacy model.

M-04 · Browser and cache policy

The built app emits no security or private-cache headers

In-memory requests to the built Nitro app returned only Content-Type for /, /login and /api/demo/ping. The reverse proxy may add headers, but that was outside scope.

Medium

Why it matters

Without framing protection, authenticated pages can be embedded for clickjacking. Without explicit private, no-store, diary/recipe HTML and server data rely on default browser/proxy caching. Missing CSP and nosniff reduce defense in depth.

Evidence

  • src/start.ts:4–10
  • vite.config.ts:6–21
  • src/routes/diary/$id.tsx:12–14
  • Built response header check: three 200 responses, only content type.

Low-friction fix

Add frame-ancestors 'none'/X-Frame-Options, nosniff, Referrer-Policy, a restrictive Permissions-Policy and no-store on authenticated responses. Set HSTS at the HTTPS edge. Roll CSP out in report-only mode first because SSR uses inline hydration scripts.

Acceptance test

Production cannot be framed; HTTPS redirects and HSTS are active; authenticated HTML/data are never stored by shared caches; and CSP report-only shows no unexpected violations before enforcement.

L-01 · CSRF defense in depth

The custom upload API is outside same-origin CSRF middleware

Server functions are origin-checked, but the cookie-authenticated multipart API is not. SameSite=Lax blocks ordinary cross-site POSTs, so this is a narrower sibling- subdomain/future-configuration risk.

Low

Why it matters

A compromised sibling subdomain is same-site and can submit with cookies. A future cookie-policy change could widen exposure. Authenticated state-changing APIs should not rely on cookie defaults alone.

Evidence

  • src/start.ts:4–6
  • src/routes/api/recipes/upload.ts:42–50

Low-friction fix

Apply the same-origin check to unsafe custom API methods, or explicitly validate Origin/Sec-Fetch-Site on this route. Keep Better Auth’s own origin/CSRF protection unchanged.

L-02 · Secret at rest

Google OAuth tokens are stored unencrypted in PostgreSQL

Better Auth token encryption is not enabled, so a database or backup read leak can expose provider tokens in addition to application data.

Low

Why it matters

Current scopes appear limited to identity/profile, reducing impact, but usable provider tokens extend a database compromise beyond Veles.

Evidence

  • src/db/schema/auth.schema.ts:39–43
  • src/lib/auth/auth.ts:10–38

Low-friction fix

Enable account.encryptOAuthTokens, keep database/backups private and encrypted, and plan auth-secret rotation before changing production token handling.

L-03 · Container hardening

The production container runs the Node process as root

The final image copies the build and starts Node without switching to an unprivileged user.

Low

Why it matters

Root is not a remote exploit by itself, but it increases the consequences of a runtime/native-library compromise inside the container.

Evidence

  • Dockerfile:22–30

Low-friction fix

Copy with --chown=node:node, add USER node, and enable no-new-privileges/capability dropping in Dokploy if compatible. Pin the base image by digest when convenient.

L-04 · Avoidable production surface

Demo pages and demo server endpoints ship publicly

Component, SSR, query and API demos are present in the generated route tree and production bundle. They expose no real secret, but add callable behavior and log noise.

Low

Why it matters

Extra routes make inventory and monitoring less clear. The component demo also ships an explicitly fake password value, which may create misleading scanner noise.

Evidence

  • src/routes/demo/components.tsx:44, 177–188
  • src/routes/demo/start.server-funcs.tsx:7–24
  • src/routes/api/demo/ping.ts:3–11

Low-friction fix

Remove demo routes from the production route tree or keep them in a separate local demo app. At minimum, return 404 for /demo/** and /api/demo/** in production.

L-05 · Log privacy

User filenames and full stacks reach production error logs

Upload failures log a client-controlled original filename and stack; general middleware logs exception messages/stacks. Bodies, cookies and authorization headers are not logged.

Low

Why it matters

Filenames often contain people, dates or private subjects. Stack traces are useful but should have short retention and restricted access in a personal-data app.

Evidence

  • src/routes/api/recipes/upload.ts:154–158, 223–227
  • src/lib/middleware/logMiddleware.ts:28–36

Low-friction fix

Omit/hash the filename, log stable error codes and request IDs, redact sensitive keys centrally, and set deliberate production retention/access. Keep generic client errors as implemented.

L-06 · Local and public-repo privacy

Local env permissions are broad, and a seed fixture contains personal identifiers

The ignored local .env is mode 0644, while the committed reset fixture includes a real-looking name, email, Google subject and avatar URL. No credential value was found tracked.

Low

Why it matters

Other local accounts can read a 0644 env file, and public identifiers aid correlation or targeted phishing. This is privacy hygiene rather than an app authorization flaw.

Evidence

  • scripts/db-reset.mjs:6–24
  • Local filesystem mode: .env = 0644
  • .gitignore:7–8 and .dockerignore:5 protect env files.

Low-friction fix

Run chmod 600 .env, use a restrictive umask, and replace the fixture identity with clearly synthetic values. If repository history is already public, decide whether removing historical personal data is worth the disruption.

L-07 · Supply chain and release gate

Checks run after a main push can already trigger production deploy

The public workflow contains no production secret or deployment step, which is good, but Actions use mutable major tags and the checks triggered on main cannot prevent Dokploy reacting to that same push.

Low

Why it matters

A bad direct push may deploy before or despite a failing post-push workflow. Mutable Action tags slightly weaken build reproducibility.

Evidence

  • .github/workflows/checks.yml:3–7, 15–23
  • .github/workflows/checks.yml:28–44

Low-friction fix

Protect main and require the existing PR check before merge; add permissions: contents: read; pin Actions to reviewed SHAs when convenient. Keep sensitive production deployment outside GitHub Actions.

05 / Verified controls

What is already working well

These controls materially reduce risk and are why no critical issue was assigned. They should be preserved while hardening the deployment.

User ownership on every sensitive query

Diary CRUD and recipe reads/rating updates combine object ID with the authenticated user ID. No current IDOR or cross-account read/write was found.

Parameterized database access

Runtime user values flow through Drizzle predicates. No user-controlled raw SQL or string-built query was found.

Boundary validation on server functions

ArkType validates UUIDs, rating range, dates and diary content limits; server functions consistently include log middleware.

No direct browser injection sink

No app use of dangerous HTML insertion, eval, JavaScript URLs or user-controlled open redirect was found. React escaping protects rendered text.

Errors are sanitized at the client boundary

Unexpected server-function errors become a generic safe error, and upload failures do not return stack, database or R2 details.

Auth defaults retain core protections

Better Auth’s own endpoints use origin/CSRF checks; cookies are HttpOnly and SameSite=Lax, and become Secure when the configured origin is HTTPS.

Uploads use server-owned keys and safe output

Object keys and ownership are generated server-side. Images are decoded and re-encoded as WebP, normally removing source metadata and active formats.

Secrets are excluded from source and images

Environment files are ignored by Git and Docker. No sensitive local env value was found in the public build, and production source maps are absent.

Dependency audit is clean

The current registry audit reported zero advisories across 548 resolved production, development and optional dependency entries on the review date.

06 / Production verification

Checks that need the live deployment

Dokploy, the public TLS edge, PostgreSQL network policy and R2/CDN settings were not accessible. Run this list against the real hostname before considering the review closed.

HTTP and authentication

  • HTTP redirects to HTTPS and HSTS is present at the edge.
  • Session cookies are Secure; HttpOnly; SameSite=Lax.
  • The deployed client bundle contains no localhost auth origin.
  • Foreign origins fail auth and custom upload requests.
  • Repeated login/signup requests return 429 using the real client IP.
  • Authenticated HTML/data return Cache-Control: private, no-store.
  • The application cannot be embedded in an unrelated iframe.

Data and infrastructure

  • Oversized request bodies are rejected before reaching Node.
  • R2 listing, CORS, lifecycle and direct-read policy match the privacy decision.
  • R2 credentials can access only the required bucket and operations.
  • The remote PostgreSQL connection requires TLS and is not publicly reachable.
  • Production auth secret is unique, random, rotated and not the example value.
  • Backups are encrypted, access-controlled and have a tested restore path.
  • Logs have short retention and do not capture cookies, bodies or personal filenames.

08 / Owner decisions

Three risks are policy choices

Security should match the actual use case. Recording these choices avoids accidentally treating a convenient default as a privacy guarantee.

Recommended for this personal app

  1. Owner-only or explicit email-allowlist registration.
  2. Private recipe photos via short-lived URLs or authenticated reads.
  3. A shorter session than the current 30 days, unless the convenience tradeoff is deliberate.

Reasonable accepted-risk alternative

  1. Public signup, but only with verified email, abuse limits and quotas.
  2. Public-by-link photos, documented as non-private and stripped of metadata.
  3. Thirty-day sessions, paired with device security and an account session-revoke UI.
